# Docker Architecture

## Overview

Docker uses a client-server architecture where the Docker client communicates with the Docker daemon, which handles building, running, and distributing containers. Understanding this architecture is essential for troubleshooting issues, optimizing performance, and working effectively with Docker in production environments. The architecture consists of several components that work together seamlessly to provide the container runtime experience.

## Prerequisites

- Understanding of what containers are (covered in previous file)
- Basic knowledge of client-server architecture
- Familiarity with command-line interfaces
- Understanding of what a daemon/service is in operating systems

## Core Concepts

### The Docker Daemon (dockerd)

The Docker daemon (dockerd) is the persistent background service that manages Docker objects such as images, containers, networks, and volumes. It listens for Docker API requests and processes them. The daemon can communicate with other daemons to manage Docker services in a distributed environment. On Linux systems, dockerd typically runs as a system service, while on Mac and Windows, it runs as a hidden application that manages a Linux virtual machine.

The daemon performs several critical functions: it pulls images from registries, builds images from Dockerfiles, manages container lifecycle (create, start, stop, delete), manages networks and volumes, and scales containers when using orchestration tools. Understanding the daemon is crucial for debugging and performance tuning.

### The Docker Client (docker)

The Docker client (docker) is the primary way most users interact with Docker. When you run commands like `docker run` or `docker build`, the client sends these commands to the dockerd daemon, which executes them. The Docker client can communicate with multiple daemons, allowing you to control both local and remote Docker hosts.

The Docker client and daemon can run on the same machine (the typical development scenario), or they can be separated. In production environments, you might have the client on your local machine while the daemon runs on a remote server. This separation allows for flexible deployment architectures.

### Docker Registries

A Docker registry is a storage and distribution system for named Docker images. The most common registry is Docker Hub (hub.docker.com), which contains many pre-built images you can pull and use. Registries can be public (anyone can pull images) or private (access controlled).

When you run `docker pull nginx:latest`, you're pulling the nginx image with the tag "latest" from Docker Hub. You can also run your own private registry using Docker Registry (a containerized registry server) or use cloud-based private registries like AWS ECR, Google Container Registry, or Azure Container Registry.

### Docker Objects

When you use Docker, you create and work with several types of objects:

**Images** are read-only templates used to create containers. An image is defined by a Dockerfile and consists of multiple layers. Images are versioned and can be tagged for easy identification.

**Containers** are runnable instances of images. You can create, start, stop, move, or delete containers. A container is isolated from other containers and the host system.

**Networks** provide communication between containers and between containers and the outside world. Docker includes several network drivers for different use cases.

**Volumes** persist data generated by containers. Volumes are the preferred mechanism for persisting data in containers.

### The Docker API

Docker provides a REST API that both the Docker CLI and other programs use to communicate with the daemon. The Docker API supports authentication and can be accessed over a Unix socket (default on Linux) or over TCP (for remote access). The API is versioned, and Docker maintains backward compatibility within each major version.

## Step-by-Step Examples

### Examining Docker Components

Let's explore the Docker architecture on your system:

```bash
# Check Docker version information (client and server)
# -a shows all information including API version
docker version -a

# View Docker daemon information
# This shows the server configuration, OS, kernel, and more
docker info

# Check which Docker socket is being used
# On Linux, this is typically /var/run/docker.sock
ls -la /var/run/docker.sock
```

### Communicating with the Docker API Directly

You can communicate with the Docker API directly using curl:

```bash
# List all containers using the Docker API
# -X GET specifies the HTTP method
# -H sets the API version header
curl -X GET -H "Content-Type: application/json" \
  "http://localhost:2375/containers/json" || \
curl -X GET --unix-socket /var/run/docker.sock \
  "http://localhost/containers/json"

# Note: TCP port 2375 is unencrypted, use 2376 with TLS in production
```

### Understanding Image Layers

Images are stored as a series of read-only layers:

```bash
# Pull an image and see layer information
# --quiet only shows the image ID after pulling
docker pull ubuntu:22.04

# Inspect the image to see its layers
# This shows the JSON configuration including layer IDs
docker image inspect ubuntu:22.04 | head -50

# History shows each layer and when it was created
# Each line represents a layer in the image
docker history ubuntu:22.04
```

## Common Mistakes

- **Accessing the Docker daemon remotely without TLS**: Exposing the Docker API over TCP without TLS is a severe security risk. Always use TLS or at minimum, use SSH tunneling.
- **Not understanding client-server communication**: Beginners sometimes expect all Docker operations to happen locally when the daemon might be remote.
- **Ignoring daemon logs**: When something goes wrong, the daemon logs often contain the answer. Check `journalctl -u docker` on Linux or the Docker Desktop logs.
- **Running Docker commands without daemon access**: If you can't connect to the Docker daemon, check if the service is running.
- **Confusing Docker client with daemon versions**: Always ensure your Docker client version is compatible with the daemon version.

## Quick Reference

| Component | Purpose |
|-----------|---------|
| dockerd | Docker daemon - runs in background, manages containers |
| docker | Docker CLI client - sends commands to daemon |
| Docker Registry | Storage for Docker images (Docker Hub, ECR, GCR) |
| Image | Read-only template for creating containers |
| Container | Running instance of an image |
| /var/run/docker.sock | Unix socket for local daemon communication |
