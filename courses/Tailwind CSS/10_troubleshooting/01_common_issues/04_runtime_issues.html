Runtime Issues

Understanding the Issue

Runtime issues occur when Tailwind CSS works in development but fails in production or browser environments. These can be related to CSS loading, JavaScript conflicts, or CDN issues.

Common Causes:

  • CDN loading order issues
  • Production build missing classes
  • JavaScript framework conflicts
  • Browser caching old CSS
  • Async/deferred script loading issues

Standard Example: The Problem

Problem: CDN Loading Race Condition

// HTML - Loading Tailwind too late
<html>
<head>
    <script src="app.js"></script>  // App loads first!
    <script src="https://cdn.tailwindcss.com"></script>
</head>
<body>
    <div class="bg-blue-500">Content</div>
    // Classes flash unstyled then apply - FART
</body>
</html>

// Problem: JavaScript modifies DOM before Tailwind is ready
// Result: Content Flashes After Render (FART)
// The blue background appears briefly without styles

Solution: Proper Loading Order

// SOLUTION 1: Load Tailwind first (in head, blocking)
<html>
<head>
    <script src="https://cdn.tailwindcss.com"></script>
    <script src="app.js"></script>
</head>

// SOLUTION 2: Use async with DOMContentLoaded
document.addEventListener('DOMContentLoaded', () => {
    // Initialize your app AFTER DOM is ready
    initApp();
});

// SOLUTION 3: For production - generate CSS at build time
// Instead of CDN, use proper build pipeline
npm install -D tailwindcss postcss autoprefixer

// Run build: npx tailwindcss -i input.css -o output.css
// Then reference output.css in HTML

Real-World Example 1: React/Framework Issue

// React + Tailwind Runtime Issue
// Problem: Using JIT with dynamic styles

// App.jsx - Problematic code
function Button({ variant }) {
    // This works in dev but may fail in prod
    const colorClass = `bg-${variant}-500`;
    
    return (
        <button className={colorClass}>
            Click
        </button>
    );
    // Error: Class not generated in production build
}

// SOLUTION 1: Use classnames library
import clsx from 'classnames';

function Button({ variant, size }) {
    return (
        <button className={clsx(
            'px-4 py-2 rounded',
            {
                'bg-blue-500': variant === 'primary',
                'bg-red-500': variant === 'danger',
                'bg-gray-500': variant === 'secondary'
            }
        )}>
            Click
        </button>
    );
}

// SOLUTION 2: Use Template Literals with conditionals
function Button({ variant }) {
    return (
        <button className={`px-4 py-2 rounded ${variant === 'primary' ? 'bg-blue-500' : 'bg-red-500'}`}>
            Click
        </button>
    );
}

Output:

Production build: Missing classes cause broken UI

Fix: Using clsx or conditional template literals works

Real-World Example 2: Browser Caching

// PROBLEM: Browser serves stale CSS
// User sees old styles after deployment

// index.html - No cache busting
<link rel="stylesheet" href="styles.css">

// Browser caches styles.css for days
// After deploy: New CSS not loaded!

// SOLUTION 1: Query string versioning
<link rel="stylesheet" href="styles.css?v=1.0.5">

// SOLUTION 2: Build tool auto-versioning
// Vite, Webpack can add content hash
// Output: styles.a1b2c3d4.css
<link rel="stylesheet" href="styles.a1b2c3d4.css">

// SOLUTION 3: Configure server headers
// .htaccess (Apache)
<FilesMatch "\.(css)$">
    Header set Cache-Control "no-cache, must-revalidate"
</FilesMatch>

// nginx.conf
location ~* \.(css)$ {
    add_header Cache-Control "no-cache, must-revalidate";
}

// SOLUTION 4: Use Tailwind CDN with version
<script src="https://cdn.tailwindcss.com?plugins=forms,typography"></script>
<script>
    tailwind.config = {
        // Your config
    }
</script>

Output:

Without cache busting: Users see outdated styles

With versioning: Fresh CSS loaded on each deploy

Edge Cases & Considerations

Edge Cases

  • • CDN fails when offline
  • • Third-party scripts may conflict
  • • iframe isolation issues
  • • Service worker caching issues

Performance

  • • CDN adds latency on first load
  • • Build pipeline is more performant
  • • Consider self-hosted CDN

Accessibility

  • • FART (flash) can trigger motion sensitivity
  • • Ensure no flash of unstyled content

Cross-Browser

  • • CDN works in all modern browsers
  • • IE11 requires polyfills
  • • Test in target browsers
