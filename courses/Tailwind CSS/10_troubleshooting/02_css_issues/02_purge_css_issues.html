PurgeCSS Issues

Understanding the Issue

PurgeCSS is integrated into Tailwind to remove unused styles, but misconfiguration can cause it to remove styles that are actually needed. This results in broken styling in production.

Common Causes:

  • Content patterns not matching source files
  • Classes used in string concatenation
  • Dynamic class names not safelisted
  • Incorrect purge configuration
  • JavaScript-generated content not scanned

Standard Example: The Problem

Problem: Too Aggressive Purging

// tailwind.config.js - Problematic config
module.exports = {
    content: ['./src/*.html'],  // Too narrow!
    // Misses: ./src/components/*.html
    // Misses: ./src/pages/*.html
    
    theme: {},
    plugins: []
}

// Result: Only classes found in ./src/*.html are kept
// All classes in components/pages are REMOVED!

// In your component:
<div class="text-red-500 bg-blue-100 p-4">
    Warning Message
</div>

// If component is in ./src/components/header.html
// Classes text-red-500, bg-blue-100, p-4 are PURGED!

Solution: Comprehensive Content Patterns

// tailwind.config.js - Better config
module.exports = {
    content: [
        './src/**/*.html',         // All HTML files
        './src/**/*.{js,ts,jsx,tsx}',  // All JS/TS files
        './components/**/*.{html,vue,svelte}',
        './pages/**/*.html'
    ],
    safelist: [
        // Always keep these classes
        'bg-blue-500',
        'text-red-500',
        'p-4',
        'flex',
        // Dynamic classes used at runtime
        /^bg-/,
        /^text-/
    ],
    theme: {},
    plugins: []
}

// safelist preserves classes even if not detected
// Regex patterns: /^bg-/ keeps ALL bg-* classes

Real-World Example 1: JavaScript Frameworks

// React component - class generated dynamically
function Alert({ type, children }) {
    // PROBLEM: This class is not detected by PurgeCSS
    const bgClass = `bg-${type}-100`;
    const textClass = `text-${type}-800`;
    
    return (
        <div className={`p-4 rounded ${bgClass} ${textClass}`}>
            {children}
        </div>
    );
}

// Usage: <Alert type="red">Warning!</Alert>
// Generates: bg-red-100 text-red-800 p-4 rounded
// But PurgeCSS sees only: ${bgClass} ${textClass}
// These are not detected!

// SOLUTION 1: Use explicit class with conditionals
function Alert({ type, children }) {
    const colors = {
        red: 'bg-red-100 text-red-800',
        green: 'bg-green-100 text-green-800',
        blue: 'bg-blue-100 text-blue-800'
    };
    
    return (
        <div className={`p-4 rounded ${colors[type]}`}>
            {children}
        </div>
    );
}

// SOLUTION 2: Add to safelist in config
module.exports = {
    content: ['./src/**/*.{js,jsx,ts,tsx}'],
    safelist: [
        'bg-red-100', 'bg-red-200', 'bg-red-300',
        'bg-green-100', 'bg-green-200', 'bg-green-300',
        'text-red-800', 'text-red-700',
        'text-green-800', 'text-green-700'
    ]
}

Output:

Without safelist: Dynamic classes purged in production

With safelist: All required classes preserved

Edge Cases & Considerations

Edge Cases

  • • Classes in imported JSON data
  • • Classes in i18n translation files
  • • Dynamic components from npm packages
  • • Classes in conditionally rendered blocks

Performance

  • • Large safelist increases CSS size
  • • Use regex patterns sparingly
  • • Consider code splitting

Accessibility

  • • Focus styles may get purged
  • • Test keyboard navigation
  • • Verify color contrast maintained

Cross-Browser

  • • PurgeCSS works with all modern browsers
  • • Test in target browser versions

Quick Fix Commands

/* View what PurgeCSS is detecting */
npx tailwindcss -i input.css -o output.css --verbose 2>&1

/* Compare dev vs prod CSS size */
wc -c input.css  (before purge)
wc -c output.css (after purge)

/* Check for missing classes in output */
grep -o "bg-red-500" output.css
grep -o "text-blue-600" output.css
