Accessibility Compliance

Understanding the Issue

Accessibility compliance ensures your Tailwind CSS projects meet established standards like WCAG (Web Content Accessibility Guidelines), Section 508, and ADA. Non-compliance can lead to legal issues, exclusions, and poor user experience for millions of people with disabilities.

Key Standards:

  • WCAG 2.1 (AA level - minimum legal requirement)
  • Section 508 (US federal government requirement)
  • ADA Title III (US public accommodations)
  • EN 301 549 (European standard)
  • CVAA (21st Century Communications Act)

Standard Example: WCAG Compliance Checklist

Problem: Non-Compliant Code

<!-- PROBLEM: Missing language attribute -->
<html>  <!-- Must include lang="en" -->

<!-- PROBLEM: Missing alt text -->
<img src="logo.png" />  <!-- Screen readers can't describe -->

<!-- PROBLEM: Missing form labels -->
<input type="email" placeholder="Email" />
<!-- Placeholder is NOT a label! -->

<!-- PROBLEM: Links with same text -->
<a href="/">Click here</a>
<a href="/about">Click here</a>
<!-- Can't distinguish in screen reader -->

<!-- PROBLEM: Auto-playing media with sound -->
<video autoplay>
    <source src="video.mp4">
</video>
<!-- Violates WCAG for audio control -->

<!-- PROBLEM: Missing document structure -->
<div class="text-3xl">Title</div>
<div class="text-2xl">Section</div>
<!-- Must use semantic heading tags -->

<!-- PROBLEM: Keyboard traps -->
<div tabindex="0" onkeydown="return false">
    <!-- User can't escape with keyboard -->
</div>

Solution: Compliant Code

<!-- SOLUTION: Proper language attribute -->
<html lang="en">

<!-- SOLUTION: Descriptive alt text -->
<img src="logo.png" alt="Company logo showing a mountain" />

<!-- SOLUTION: Proper form labeling -->
<label for="email">Email address</label>
<input 
    type="email" 
    id="email" 
    placeholder="you@example.com"
    aria-describedby="email-hint"
/>
<span id="email-hint" class="text-gray-500">
    We'll never share your email
</span>

<!-- SOLUTION: Unique link text -->
<a href="/">Go to homepage</a>
<a href="/about">Learn about us</a>

<!-- SOLUTION: Audio control -->
<video>
    <source src="video.mp4">
    <track kind="captions" src="captions.vtt">
</video>
<button onclick="togglePlay()">Play/Pause</button>

<!-- SOLUTION: Proper heading structure -->
<h1>Main Title</h1>
<h2>Section Heading</h2>
<h3>Subsection</h3>

<!-- SOLUTION: Keyboard navigable -->
<dialog>
    <button autofocus>Close</button>
    <!-- User can tab out with Escape -->
</dialog>

Real-World Example 1: Navigation Accessibility

<!-- PROBLEM: Inaccessible navigation -->
<nav class="flex gap-4">
    <!-- All links without semantic container -->
    <a href="/">Home</a>
    <a href="/products">Products</a>
    <a href="/about">About</a>
    <a href="/contact">Contact</a>
    
    <!-- Screen reader announces: "Home, link, Products, link..." -->
    <!-- No way to skip to main content -->
</nav>

<!-- SOLUTION: Accessible navigation -->
<nav aria-label="Main navigation">
    <ul class="flex gap-4" role="list">
        <li>
            <a href="/">Home</a>
        </li>
        <li>
            <a href="/products">Products</a>
        </li>
        <li>
            <a href="/about">About</a>
        </li>
        <li>
            <a href="/contact">Contact</a>
        </li>
    </ul>
</nav>

<!-- Skip link for keyboard users -->
<a href="#main-content" class="sr-only focus:not-sr-only">
    Skip to main content
</a>

<!-- Main content with landmark -->
<main id="main-content">
    <h1>Welcome to our website</h1>
</main>

<!-- Current page indicator -->
<nav aria-label="Main navigation">
    <ul>
        <li>
            <a href="/" aria-current="page">Home</a>
        </li>
    </ul>
</nav>

Output:

Screen reader announces: "Main navigation, navigation landmark, list with 4 items"

Skip link allows keyboard users to bypass navigation

aria-current="page" indicates current location

Real-World Example 2: Modal/Dialog Accessibility

<!-- PROBLEM: Inaccessible modal -->
<div class="modal">
    <h2>Confirm Action</h2>
    <p>Are you sure?</p>
    <button onclick="close()">X</button>
    <button onclick="confirm()">Confirm</button>
</div>
<!-- Issues: -->
<!-- - No aria-modal="true" -->
<!-- - No focus trapping -->
<!-- - NoEscape to close -->
<!-- - X button unlabeled -->
<!-- - Not announced as dialog -->

<!-- SOLUTION: Fully accessible modal -->
<dialog 
    id="confirm-modal"
    aria-modal="true"
    aria-labelledby="modal-title"
    aria-describedby="modal-desc"
>
    <h2 id="modal-title">Confirm Action</h2>
    <p id="modal-desc">Are you sure you want to proceed?</p>
    
    <button 
        onclick="closeModal()"
        aria-label="Close dialog"
    >×</button>
    
    <button onclick="confirm()">Confirm</button>
</dialog>

<!-- JavaScript for focus management -->
<script>
function openModal() {
    const modal = document.getElementById('confirm-modal');
    modal.showModal();
    
    // Focus first input or button in modal
    const focusable = modal.querySelector('button, input, [tabindex]');
    if (focusable) focusable.focus();
    
    // Prevent background scroll
    document.body.style.overflow = 'hidden';
}

function closeModal() {
    const modal = document.getElementById('confirm-modal');
    modal.close();
    
    // Return focus to trigger button
    triggerButton.focus();
    
    // Restore scroll
    document.body.style.overflow = '';
}

// Handle Escape key
modal.addEventListener('cancel', (e) => {
    e.preventDefault();
    closeModal();
});
</script>

Output:

Screen reader announces: "Dialog, Confirm Action, modal"

Focus trapped inside modal while open

Escape key closes modal

Focus returns to trigger on close

WCAG 2.1 Guidelines Summary

┌─────────────────────────────────────────────────────────────┐
│                 WCAG 2.1 PRINCIPLES                            │
├─────────────────────────────────────────────────────────────┤
│  1. PERCEIVABLE                                              │
│     └─ Info must be presentable to users                    │
│        - Text alternatives for images                      │
│        - Captions for video                                │
│        - Color not sole indicator                          │
├─────────────────────────────────────────────────────────────┤
│  2. OPERABLE                                                │
│     └─ Interface must be usable                           │
│        - Keyboard accessible                              │
│        - Enough time to read                              │
│        - No seizures (3 flashes/second max)              │
│        - Navigation hints                                │
├──────────────────────��─��────────────────────────────────────┤
│  3. UNDERSTANDABLE                                         │
│     └─ Content must be understandable                       │
│        - Readable language                                │
│        - Predictable behavior                            │
│        - Error identification                            │
├─────────────────────────────────────────────────────────────┤
│  4. ROBUST                                                 │
│     └─ Content must be interpreted reliably               │
│        - Valid HTML                                       │
│        - ARIA where needed                               │
│        - Name, Role, Value                               │
└─────────────────────────────────────────────────────────────┘

Edge Cases & Legal Considerations

Legal Cases

  • Domino's Pizza v. NLRB (2019)
  • CVAA penalties increasing
  • ADA lawsuits up 300%+
  • WCAG 2.0 AA is defense standard

Mobile Considerations

  • Touch targets 44x44px minimum
  • VoiceOver/TalkBack testing
  • Portrait/landscape orientation
  • Small screen compatibility

Testing Tools

  • axe DevTools (Chrome/Firefox)
  • WAVE Web Accessibility Tool
  • Google Lighthouse
  • NVDA Screen Reader

Performance

  • Accessibility testing adds minimal time
  • Automated tools catch ~40% issues
  • Manual testing essential
  • User testing ideal
