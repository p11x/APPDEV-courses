WordPress + Tailwind CSS Integration

Overview

Integrating Tailwind CSS with WordPress can be done through several approaches: using a custom theme with Tailwind built assets, installing Tailwind via a plugin, or enqueuing compiled Tailwind CSS in your theme's functions.php. The most flexible approach involves setting up a build process with npm, PostCSS, and Tailwind CLI.

Key Point: WordPress + Tailwind integration requires understanding WordPress enqueuing system and building a proper asset pipeline. The key is proper file paths in functions.php and purge configuration for production.

Standard Example: Theme Setup with NPM

Setting up Tailwind CSS in a custom WordPress theme using npm and PostCSS:

// package.json

{

"name": "my-tailwind-theme",

"devDependencies": {

"tailwindcss": "^3.4.0",

"autoprefixer": "^10.4.0",

"postcss": "^8.4.0"

}

}

// tailwind.config.js

module.exports = {

content: ["./**/*.php", "./**/*.js"],

theme: {

extend: {},

},

plugins: [],

};

// postcss.config.js

module.exports = {

plugins: {

tailwindcss: {},

autoprefixer: {},

}

};

// src/style.css (input file)

@tailwind

base;

@tailwind

components;

@tailwind

utilities;

// build command: npx tailwindcss -i ./src/style.css -o ./build/style.css

Real-World Example 1: Functions.php Enqueue

Properly enqueuing Tailwind CSS in WordPress theme functions.php:

// functions.php - Enqueue Tailwind CSS

function

register_theme_assets

() {

// Register the compiled Tailwind CSS

wp_register_style

(

'tailwind-main',

get_template_directory_uri() . '/build/style.css',

[],

'1.0.0',

'all'

);

 

// Enqueue for frontend

wp_enqueue_style

(

'tailwind-main'

);

}

 

// Hook into wp_enqueue_scripts

add_action

(

'wp_enqueue_scripts',

'register_theme_assets'

);

 

// Add inline styles for custom theme colors

function

custom_theme_colors

() {

$primary_color = get_theme_mod('primary_color', '#3b82f6');

$css = ".btn-primary { background-color: {$primary_color}; }";

wp_add_inline_style

(

'tailwind-main',

$css

);

}

add_action

(

'wp_enqueue_scripts',

'custom_theme_colors'

);

Edge Cases & Considerations

  • Dynamic Content: WordPress content is database-driven; ensure content paths include all template files for PurgeCSS.
  • Gutenberg Blocks: When using Gutenberg, Tailwind classes in block templates must be included in content paths.
  • Page Builders: Elementor/Divi use dynamic CSS; configure safelist for classes used by page builder widgets.
  • Plugin Conflicts: Some plugins add inline styles; you may need to use !important or specific targeting.
  • RTL Support: Configure tailwind.config.js with rtl plugins for right-to-left languages.

Performance Considerations

  • Production Build: Always run production build with minification and purging enabled.
  • JIT Mode: Use Just-In-Time compiler for smaller CSS output (only used classes).
  • Cache Plugins: Ensure cache plugins are compatible with your Tailwind setup.
  • Asset Loading: Load Tailwind only on pages where needed using conditional tags.
  • Content Detection: Include all PHP template files in content configuration for proper purging.

Accessibility Notes

  • Focus States: Always include focus ring classes for keyboard navigation.
  • Skip Links: Add skip link with sr-only class for screen reader users.
  • Color Contrast: Ensure text colors meet WCAG AA standards (4.5:1 ratio).
  • Reduced Motion: Use motion-reduced classes respecting prefers-reduced-motion.
  • Semantic HTML: Use proper heading hierarchy and ARIA labels where needed.

Cross-Browser Compatibility

  • Browser Support: Tailwind v3+ requires modern browsers with CSS custom properties.
  • Autoprefixer: PostCSS with autoprefixer handles vendor prefixes automatically.
  • IE11: For legacy browser support, add additional polyfills or use Tailwind v2.
  • Flexbox/Grid: Both are well-supported; use Autoprefixer for older browser fallbacks.
