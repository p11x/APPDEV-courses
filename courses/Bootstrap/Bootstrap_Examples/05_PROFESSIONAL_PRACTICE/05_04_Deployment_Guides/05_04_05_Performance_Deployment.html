1 · Core Web Vitals

Core Web Vitals are three user-centric performance metrics Google uses as ranking signals. Optimising them is the foundation of any performance-focused deployment.

LCP
Largest Contentful Paint

Measures loading performance — the time until the largest visible element finishes rendering.

RatingValue
Good≤ 2.5 s
Needs Improvement≤ 4.0 s
Poor> 4.0 s
FID / INP
Interaction to Next Paint

Measures interactivity — how quickly the page responds to user input (click, tap, key).

RatingValue
Good≤ 200 ms
Needs Improvement≤ 500 ms
Poor> 500 ms
CLS
Cumulative Layout Shift

Measures visual stability — how much the page layout moves after initial render.

RatingValue
Good≤ 0.10
Needs Improvement≤ 0.25
Poor> 0.25
Tip: Use the web.dev/vitals documentation to stay current with metric definitions.

2 · Lighthouse Optimization

Lighthouse (built into Chrome DevTools) audits Performance, Accessibility, Best Practices, SEO, and PWA. Target a score of 90+ in every category for production.

96
100
Lighthouse CI Configuration
// lighthouserc.js — automate Lighthouse in CI/CD pipelines
module.exports = {
  ci: {
    collect: {
      startServerCommand: 'npm run preview',
      startServerReadyPattern: 'ready in',
      url: ['http://localhost:4173/'],
      numberOfRuns: 3,
    },
    assert: {
      assertions: {
        'categories:performance': ['error', { minScore: 0.9 }],
        'categories:accessibility': ['error', { minScore: 0.9 }],
      },
    },
    upload: {
      target: 'temporary-public-storage',
    },
  },
};

3 · Lazy Loading

Lazy loading defers the download of off-screen images, iframes, and videos until the user scrolls near them, reducing initial page weight and speeding up LCP.

Native loading="lazy"

The simplest approach — add loading="lazy" to <img> and <iframe> tags. The browser handles scheduling automatically.

Intersection Observer

For fine-grained control (fade-in animations, analytics triggers), use the IntersectionObserver API to load resources only when they enter the viewport.

Native Lazy Loading
<!-- loading="lazy" — browser defers fetch until near viewport -->
<img
  src="hero-large.webp"
  alt="Performance dashboard"
  width="1200"
  height="630"
  loading="lazy"
  decoding="async"
  fetchpriority="low"
  class="img-fluid rounded"
/>

<!-- fetchpriority="high" on the LCP hero image -->
<img
  src="hero-lcp.webp"
  fetchpriority="high"
  loading="eager"
  alt=""
/>
Intersection Observer Pattern
// Defer image loading with IntersectionObserver
const observer = new IntersectionObserver((entries) => {
  entries.forEach((entry) => {
    if (entry.isIntersecting) {
      const img = entry.target;
      img.src = img.dataset.src;            // swap data-src → src
      img.classList.add('loaded');
      observer.unobserve(img);              // stop observing
    }
  });
}, { rootMargin: '200px' });            // load 200 px early

document.querySelectorAll('img[data-src]')
  .forEach((img) => observer.observe(img));

4 · Critical CSS

Critical CSS is the minimal set of styles required to render above-the-fold content. Inlining it eliminates a render-blocking round-trip, improving First Contentful Paint (FCP).

Place critical CSS inside a <style> tag in the <head>. Then load the full stylesheet asynchronously: 

<!-- Inline critical CSS -->
<style>
  /* only above-the-fold rules */
  body { margin: 0; font-family: system-ui; }
  .hero { min-height: 100vh; }
</style>

<!-- Load full CSS async — onload swaps to "stylesheet" -->
<link rel="preload" as="style"
      href="full.css"
      onload="this.rel='stylesheet'" />

Automated tools extract critical CSS from your rendered HTML so you don't have to maintain it manually:

  • critical — npm package by Addy Osmani
  • Penthouse — headless Chromium-based extraction
  • critters — webpack/Next.js plugin
# Install critical globally
npm install -g critical

# Extract and inline critical CSS
critical https://example.com \
  --base . \
  --inline \
  --css dist/css/main.css \
  --dimensions 320x568 1200x900

Preload the full CSS so it's fetched in parallel with the inlined critical styles, then applied without a flash of unstyled content (FOUC). 

<!-- Preload full CSS, then switch to stylesheet -->
<link rel="preload" as="style"
      href="/css/main.min.css"
      onload="this.onload=null;this.rel='stylesheet'" />

<!-- Noscript fallback for JS-disabled browsers -->
<noscript><link rel="stylesheet"
      href="/css/main.min.css" /></noscript>

5 · Resource Hints

Resource hints tell the browser which origins or assets to prioritise, reducing DNS look-up, TLS handshake, and connection times.

Establishes a full TCP + TLS handshake to a third-party origin before the browser discovers a resource that needs it. 

<link rel="preconnect"
      href="https://cdn.jsdelivr.net"
      crossorigin />

Lightweight fallback for older browsers — only resolves DNS, does not open a full connection. 

<link rel="dns-prefetch"
      href="https://fonts.googleapis.com" />

Forces the browser to download a resource immediately at high priority — use sparingly for LCP assets. 

<link rel="preload"
      href="/fonts/inter-var.woff2"
      as="font"
      type="font/woff2"
      crossorigin />

Low-priority hint that downloads a resource likely needed for the next navigation. Good for SPA route chunks. 

<link rel="prefetch"
      href="/js/dashboard.chunk.js"
      as="script" />

6 · Image Optimization

Images often account for 50 %+ of page weight. Modern formats (WebP, AVIF), responsive <picture>, and explicit dimensions dramatically reduce transfer size and CLS.

Format Compression Transparency Animation Browser Support
WebP 25-34 % smaller than JPEG Yes Yes 97 %+
AVIF ~50 % smaller than JPEG Yes Yes 90 %+
JPEG Baseline No No 100 %
PNG Lossless, large Yes No 100 %
Responsive <picture> Element
<picture>
  <!-- AVIF for supporting browsers (smallest) -->
  <source
    srcset="hero-800.avif 800w, hero-1200.avif 1200w"
    sizes="(max-width: 768px) 100vw, 1200px"
    type="image/avif" />

  <!-- WebP fallback -->
  <source
    srcset="hero-800.webp 800w, hero-1200.webp 1200w"
    sizes="(max-width: 768px) 100vw, 1200px"
    type="image/webp" />

  <!-- JPEG ultimate fallback -->
  <img
    src="hero-1200.jpg"
    alt="Performance dashboard screenshot"
    width="1200"   /* explicit width prevents CLS */
    height="630"    /* explicit height prevents CLS */
    loading="lazy"   /* native lazy load */
    decoding="async" /* decode off main thread */
    class="img-fluid rounded shadow"
  />
</picture>
CLS warning: Always set explicit width and height on <img> tags. Browsers use these to reserve space and prevent layout shifts.

7 · Performance Budgets

A performance budget sets concrete limits on bundle size, number of requests, and metric targets. If a PR exceeds the budget, CI fails and the change is rejected.

Budget Types
  • Size budgets — total JS, CSS, image, and font file sizes (e.g. JS ≤ 170 KB compressed).
  • Quantity budgets — max number of HTTP requests per page.
  • Metric budgets — LCP ≤ 2.5 s, CLS ≤ 0.1, TBT ≤ 200 ms.
Enforcement Tools
  • Webpack — performance.maxAssetSize
  • Bundlewatch — tracks sizes across PRs
  • Lighthouse CI — metric assertions in CI pipeline
  • Size Limit — GitHub Action for JS budgets
Webpack Performance Configuration
// webpack.config.js — enforce size budgets at build time
module.exports = {
  performance: {
    hints: 'error',           // 'warning' | 'error' | false
    maxAssetSize: 170 * 1024,  // 170 KB per asset
    maxEntrypointSize: 300 * 1024, // 300 KB total entry
    assetFilter: (asset) =>
      /* only check JS & CSS bundles */
      /\.(js|css)$/.test(asset),
  },
};
Size Limit (GitHub Action)
// .size-limit.json
[
  {
    "name": "Main JS bundle",
    "path": "dist/main.*.js",
    "limit": "170 KB",
    "gzip": true
  },
  {
    "name": "CSS bundle",
    "path": "dist/style.*.css",
    "limit": "50 KB",
    "gzip": true
  }
]
CI Workflow (GitHub Actions)
# .github/workflows/perf.yml
name: Performance Budget
on: [pull_request]
jobs:
  budget:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: '20'
      - run: npm ci
      - run: npm run build
      - uses: andresz1/size-limit-action@v1
        with:
          github_token: ${{ secrets.GITHUB_TOKEN }}
