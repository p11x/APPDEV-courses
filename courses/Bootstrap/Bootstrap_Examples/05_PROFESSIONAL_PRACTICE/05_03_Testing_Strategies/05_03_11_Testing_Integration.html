Docker Compose Setup
# docker-compose.test.yml
version: '3.8'
services:
  app:
    build: .
    ports: ["3000:3000"]
    depends_on:
      db:
        condition: service_healthy
  db:
    image: postgres:16
    environment:
      POSTGRES_DB: testdb
    healthcheck:
      test: ["CMD", "pg_isready"]
Key Considerations
  • Use isolated test databases
  • Wait for services to be healthy
  • Use fixed ports for predictability
  • Configure test-specific env vars

// tests/integration/api.test.js
describe('Product API Integration', () => {
  beforeAll(async () => {
    await seedDatabase();
  });

  afterAll(async () => {
    await cleanupDatabase();
  });

  test('GET /api/products returns list', async () => {
    const res = await fetch('http://localhost:3000/api/products');
    expect(res.status).toBe(200);
    const data = await res.json();
    expect(data).toHaveLength(2);
  });

  test('POST /api/products creates item', async () => {
    const res = await fetch('http://localhost:3000/api/products', {
      method: 'POST',
      headers: { 'Content-Type': 'application/json' },
      body: JSON.stringify({ name: 'Widget C', price: 19.99 })
    });
    expect(res.status).toBe(201);
  });
});

Integration Testing Strategies

Big Bang Integration

All modules integrated simultaneously and tested as a complete system.

Simple Hard to Debug
Complexity: 70%
Debug Ease: 40%
Incremental Integration

Modules integrated one by one, tested incrementally. Top-down or bottom-up approach.

Easy Debug More Time
Complexity: 30%
Debug Ease: 90%
Strategy Approach Pros Cons Best For
Top-Down Test high-level modules first with stubs Early interface validation Needs stubs for lower modules UI-heavy applications
Bottom-Up Test low-level modules first with drivers No stubs needed UI tested last Data-intensive apps
Sandwich Combines top-down and bottom-up Best of both Complex management Large enterprise systems
Continuous Test as modules are completed Early bug detection Requires discipline Agile teams
