Cross-Browser Performance

Optimization techniques for consistent performance across Chrome, Firefox, Safari, and Edge.

1. CSS Performance Patterns

Efficient CSS Selectors

Selectors are evaluated right-to-left. Prefer specific, fast selectors:

/* FAST - Class selector */
.nav-link { }
.btn-primary { }

/* SLOWER - Descendant combinator */
.nav .nav-link { }

/* SLOWEST - Universal + deep nesting */
div > * > .class { }

/* Avoid in performance-critical CSS */
:not() /* with complex selectors */
:has() /* expensive in large DOMs */
CSS Containment

Isolate rendering scope for browser optimization:

/* Full containment - most restrictive */
.card {
  contain: layout style paint;
}

/* Size containment - skip size calc */
.virtual-list-item {
  contain: size;
}

/* Layout containment - isolate layout */
.sidebar {
  contain: layout;
}

/* content-visibility: auto */
/* Skips rendering off-screen content */
.below-fold {
  content-visibility: auto;
  contain-intrinsic-size: 0 300px;
}
will-change Property

Hint to browser about upcoming changes. Use sparingly!

Hover me (will-change: transform) 
/* DO: Apply to elements that will animate */
.animated-element {
  will-change: transform, opacity;
}

/* DON'T: Apply to too many elements */
/* Each will-change creates a new layer,
   consuming GPU memory */
* { will-change: all; } /* NEVER DO THIS */
Avoid Layout Thrashing

Reading and writing layout properties in a loop forces repeated reflows:

// BAD: Layout thrashing
elements.forEach(el => {
  const height = el.offsetHeight; // READ
  el.style.height = height + 10 + 'px'; // WRITE
  const width = el.offsetWidth; // READ (reflow!)
  el.style.width = width + 10 + 'px'; // WRITE
});

// GOOD: Batch reads, then writes
const heights = elements.map(
  el => el.offsetHeight // All READs
);
const widths = elements.map(
  el => el.offsetWidth // All READs
);
elements.forEach((el, i) => {
  el.style.height = heights[i] + 10 + 'px';
  el.style.width = widths[i] + 10 + 'px';
}); // All WRITEs

2. Animation Performance

GPU-Accelerated Properties: Only transform, opacity, filter, and clip-path are composited (GPU-rendered). Animating width, height, top, left, margin triggers layout/paint and is expensive.
GPU vs CPU Animation Comparison
GPU: transform + opacity
Click to animate

Composited - no layout/paint

CPU: width + height (avoid!)
Click to animate

Triggers layout + paint (expensive)

CSS vs JavaScript Animations
Aspect CSS Transitions/Animations JavaScript (Web Animations API)
Performance Can be compositor-only Main thread unless compositor
Control Limited (start/end, keyframes) Full control (pause, reverse, etc.)
Scroll-linked Not natively ScrollTimeline support
Browser Support Universal Web Animations API not in all browsers
Use Case Simple state transitions Complex, interactive animations

Lazy Loading Images

Native lazy loading with loading="lazy" is supported in all modern browsers. No JavaScript needed. 

<!-- Native lazy loading -->
<img src="photo.jpg" loading="lazy" alt="..." 
     width="400" height="300">

<!-- Always set width/height to prevent layout shift -->
<!-- loading="lazy" should NOT be used on above-fold images -->
<img src="hero.jpg" loading="eager" alt="...">
