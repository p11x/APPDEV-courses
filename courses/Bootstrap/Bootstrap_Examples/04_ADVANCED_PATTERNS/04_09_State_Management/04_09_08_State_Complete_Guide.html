3. Decision Matrix

Which pattern to choose based on your requirements.

Requirement Recommended Pattern Reason
Simple toggle (on/off) CSS Class Zero JS state needed, CSS handles it
Component-specific state Data Attribute State lives on the element, easy to read
Multi-field form JS State Object Complex state needs central management
Workflow (draft/review/publish) State Machine Strict transitions prevent invalid states
Cross-component communication Pub/Sub Decouples components, scalable
Undo/redo functionality Command Encapsulates actions with undo logic
State persistence localStorage Survives page reloads
High-frequency updates Batch/Throttle Prevents excessive re-renders
Screen reader users ARIA Live Announces state changes to AT
Mobile/touch interactions Touch State Handles touch events and gestures

4. Best Practices & Anti-Patterns

Best Practices
  • ✓ Single source of truth — one place holds each state value
  • ✓ Immutable updates — create new state objects, don't mutate
  • ✓ Validate transitions — use state machines for strict flows
  • ✓ Announce to screen readers — use aria-live for dynamic state
  • ✓ Persist user preferences — save important state to localStorage
  • ✓ Batch DOM updates — reduce re-renders for performance
  • ✓ Test state logic — unit test state transitions in isolation
  • ✓ Handle offline — queue actions when network is unavailable
Anti-Patterns
  • ✗ Scattered state — state spread across many DOM elements
  • ✗ Direct DOM reads as state — reading input.value everywhere
  • ✗ Render on every keystroke — no debouncing on input handlers
  • ✗ Color-only state indicators — inaccessible to colorblind users
  • ✗ Silent state changes — no announcements for screen readers
  • ✗ Ignoring offline — assuming always-on network connectivity
  • ✗ God state object — one massive object holding all app state
  • ✗ Unvalidated transitions — allowing any state to transition to any state
