Display Utilities — Accessibility

How display utilities affect accessibility and best practices for creating inclusive interfaces.

1. d-none vs .visually-hidden

Screen Reader Test
d-none — NOT accessible

Visible text here.

This hidden text is NOT read by screen readers because d-none removes it from the accessibility tree.

The hidden span with d-none is invisible to assistive technology.

.visually-hidden — IS accessible

Visible text here.

This hidden text IS read by screen readers because visually-hidden keeps it in the accessibility tree.

The hidden span with .visually-hidden IS visible to assistive technology.

Rule: Use .visually-hidden for content that should be available to screen readers but hidden visually. Use d-none only when you want to hide content from EVERYONE (including screen readers).

2. ARIA Attributes with Display Toggling

Always pair display changes with appropriate ARIA attributes.

Expandable Section with ARIA
Details Section

This content is now visible. The button's aria-expanded attribute has been updated to "true" to inform screen readers that this section is open.

Always use aria-expanded, aria-controls, and role attributes when toggling visibility.

JavaScript updates aria-expanded on the button every time d-none is toggled on the content.

3. Accessible Error Messages

Form errors must be accessible to screen readers.

Form with Accessible Errors
Enter a valid email address.
Please enter a valid email address (e.g., user@example.com).

The error message uses role="alert" and aria-live="assertive" so screen readers announce it immediately when it appears (d-none is removed).

4. Focus Management

Move focus appropriately when showing/hiding content.

Modal-like Pattern
Information Panel

When this panel opens, focus is moved here. When it closes, focus returns to the trigger button.

JavaScript moves focus to the panel when opened, and back to the trigger button when closed. This ensures keyboard users aren't lost after display changes.

5. Color Contrast Awareness

Ensure visible content meets WCAG contrast requirements.

Good Contrast

Black on white: 21:1 ratio ✓

White on Bootstrap primary: 4.56:1 ✓

Poor Contrast

Light gray on white: 2.32:1 ✗

Yellow on white (hidden on mobile): 1.07:1 ✗

When using display utilities to show/hide content across breakpoints, verify that visible states maintain 4.5:1 contrast (WCAG AA).

8. Display Utilities Accessibility Checklist

Check Guideline
Use .visually-hidden instead of d-none for SR-needed content
Update aria-expanded when toggling visibility
Use aria-controls to associate triggers with targets
Manage focus: move to shown content, return to trigger when hidden
Use aria-live regions for dynamic content changes
Add aria-label or aria-labelledby to interactive elements
Ensure visible content meets WCAG contrast ratios (4.5:1 AA)
Don't rely solely on display changes — pair with ARIA
