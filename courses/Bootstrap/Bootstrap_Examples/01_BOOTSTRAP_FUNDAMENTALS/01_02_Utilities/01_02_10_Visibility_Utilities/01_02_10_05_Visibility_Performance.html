Visibility Utilities - Performance

Understanding the performance implications of different visibility approaches.

1. display:none vs visibility:hidden Performance

Property Class Layout Paint Reflow Cost
display: none .d-none Removed from flow Not painted Higher (triggers reflow)
visibility: hidden .invisible Preserves space Not painted Lower (no reflow)
opacity: 0 (custom) Preserves space Painted (transparent) GPU accelerated
Key insight: Toggling .d-none triggers a reflow (layout recalculation) because it changes the document flow. Toggling .invisible only affects painting, which is cheaper.

3. CSS Containment

Use contain CSS property to limit the scope of reflows caused by visibility changes.

/* Contain layout and paint operations */
.togglable-section {
    contain: layout style paint;
}

6. Performance Comparison

Approach Trigger Reflow Trigger Repaint Rendered Best For
.d-none Yes Yes No Completely removing content
.invisible No Yes Space reserved Preventing layout shift
opacity: 0 No GPU Yes (transparent) Fade animations
clip-path No Minimal Clipped Complex reveal animations
content-visibility Deferred Deferred Deferred Off-screen content

7. Performance Best Practices

When an element is toggled frequently (e.g., tooltips, hover states), use .invisible instead of .d-none to avoid layout thrashing.

When toggling multiple elements, apply all class changes in a single JavaScript frame. Use requestAnimationFrame if needed.

Apply contain: layout to containers whose children's visibility changes frequently. This limits reflow scope.

Don't toggle .d-none on scroll events. Use IntersectionObserver instead for lazy loading and scroll-based visibility.
