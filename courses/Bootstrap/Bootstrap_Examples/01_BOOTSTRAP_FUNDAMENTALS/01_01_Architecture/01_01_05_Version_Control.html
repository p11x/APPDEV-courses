.gitignore Configuration

What to Ignore

A proper .gitignore prevents committing files that should not be in version control. These include build outputs, dependencies, and sensitive configuration.

Dependencies
node_modules/ bower_components/ vendor/

Dependencies are installed via npm install. They can be rebuilt from package-lock.json, so they should never be committed.

Build Output
dist/ build/ *.min.css *.min.js

Compiled/built files are generated from source. Commit the source, not the output. Rebuild with npm run build.

Environment & Secrets
.env .env.local .env.*.local *.pem

Environment files often contain API keys and secrets. Never commit credentials to version control.

Editor & OS Files
.vscode/ .idea/ .DS_Store Thumbs.db *.log
Sample .gitignore
# Dependencies
node_modules/
package-lock.json
# Build output
dist/
build/
css/bootstrap*.css
css/bootstrap*.css.map
# Environment
.env
.env.local
# IDE
.vscode/
.idea/
*.swp
# OS
.DS_Store
Thumbs.db
# Logs
*.log
npm-debug.log*
Tip: Use gitignore.io to generate .gitignore files for your specific stack.

Branching Strategy

Git Flow for Bootstrap Projects

A common branching model uses multiple branch types for different purposes:

main - Production-ready code
develop - Integration branch for features
feature/navbar-redesign - New feature work
feature/add-dark-mode - Another feature
release/v1.2.0 - Release preparation
hotfix/fix-modal-z-index - Urgent production fix
Branch Naming Convention

Use the format: type/short-description

Prefix Use For Example
feature/ New functionality feature/user-auth
bugfix/ Bug fixes bugfix/fix-navbar-toggle
hotfix/ Urgent production fixes hotfix/security-patch
release/ Release preparation release/v2.0.0
docs/ Documentation only docs/update-readme
Branch Commands
# Create & switch
git checkout -b feature/new-theme
# List branches
git branch -a
# Switch branch
git switch develop
# Delete branch
git branch -d feature/done
# Merge branch
git merge feature/new-theme
# Rebase (rewrite)
git rebase develop
# Push branch
git push -u origin feature/new-theme

Commit Conventions

Conventional Commits

The Conventional Commits specification provides a lightweight convention on top of commit messages.

type(scope): description
# Examples:
feat(navbar): add dark mode toggle
fix(modal): correct z-index stacking
docs(readme): update installation steps
style: format SCSS with prettier
refactor(grid): simplify column calc
test(button): add unit tests
chore: update dependencies
Commit Types

Standard types for conventional commits:

Type Purpose
feat New feature
fix Bug fix
docs Documentation changes
style Code style (no logic change)
refactor Code refactoring
test Adding/fixing tests
chore Maintenance tasks
perf Performance improvements
Breaking changes: Add ! after type: feat!: remove deprecated API

Common Workflows

Starting a New Feature
# 1. Update your local main
git checkout main
git pull origin main
# 2. Create feature branch
git checkout -b feature/custom-theme
# 3. Do your work...
git add .
git commit -m "feat(theme): add custom color scheme"
# 4. Push and create PR
git push -u origin feature/custom-theme
Syncing with Upstream
# 1. Fetch latest changes
git fetch origin
# 2. Rebase your branch
git rebase origin/main
# 3. Resolve conflicts if any
git add resolved-file.html
git rebase --continue
# 4. Force-push rebased branch
git push --force-with-lease
Use --force-with-lease instead of --force. It prevents overwriting others' work.
What to Track vs Ignore
Commit These
  • Source files (.scss, .js, .html)
  • package.json (not package-lock.json)
  • Configuration files (.browserslistrc)
  • Documentation (README, docs/)
  • .gitignore
  • CI/CD config (.github/workflows/)
Don't Commit These
  • node_modules/
  • dist/ or build/ directories
  • .env files with secrets
  • IDE configuration (.vscode/)
  • OS files (.DS_Store, Thumbs.db)
  • Log files (*.log)
