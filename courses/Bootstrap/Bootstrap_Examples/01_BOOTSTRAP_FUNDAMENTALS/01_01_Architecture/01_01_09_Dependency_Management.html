Bootstrap's Dependencies

Dependency Types

npm packages have different dependency types, each serving a different purpose.

bootstrap@5.3.3
peerDependencies
@popperjs/core ^2.11.8

Required at runtime but YOU install it. Bootstrap needs Popper.js for tooltips and dropdowns.

devDependencies
sass, postcss, autoprefixer...

Only needed during development/build. Not installed when users run npm install bootstrap.

No runtime deps
Bootstrap has zero!

Bootstrap's CSS has no dependencies at all. Only the JS bundle optionally needs Popper.js.

package.json
{
  "name": "bootstrap",
  "version": "5.3.3",
  "peerDependencies": {
    "@popperjs/core": "^2.11.8"
  },
  "devDependencies": {
    "sass": "^1.77.0",
    "postcss": "^8.4.38",
    "autoprefixer": "^10.4.19",
    "clean-css-cli": "^5.6.3",
    "terser": "^5.31.0"
  }
}

NPM Commands

Installing Bootstrap
# Install Bootstrap (CSS-only is fine)
npm install bootstrap
# Install with Popper.js for JS components
npm install bootstrap @popperjs/core
# Install specific version
npm install bootstrap@5.3.3
# Install as dev dependency
npm install --save-dev bootstrap
# Check installed version
npm list bootstrap
# Check for updates
npm outdated bootstrap
Updating & Removing
# Update to latest compatible version
npm update bootstrap
# Update to latest major version
npm install bootstrap@latest
# Remove Bootstrap
npm uninstall bootstrap
# View all installed packages
npm list --depth=0
# Clean install (reinstall everything)
rm -rf node_modules
npm install

Semantic Versioning

SemVer Format: MAJOR.MINOR.PATCH
5 . 3 . 3
MAJOR
Breaking changes MINOR
New features PATCH
Bug fixes
Change Version Bump Safe to Update?
Bug fix 5.3.3 → 5.3.4 Yes
New feature 5.3.3 → 5.4.0 Usually
Breaking change 5.3.3 → 6.0.0 Test first
Version Ranges in package.json

The symbols before version numbers control what updates are allowed:

Symbol Example Allows
^ ^5.3.3 5.x.x (minor + patch)
~ ~5.3.3 5.3.x (patch only)
≥ >=5.0.0 5.0.0 or higher
* * Any version
(exact) 5.3.3 Only 5.3.3
^ (caret) is the npm default and recommended for most cases. It allows bug fixes and new features but not breaking changes.

Security Auditing

Running Security Audits

npm audit checks your dependencies against the npm security advisory database.

# Check for vulnerabilities
npm audit
# Auto-fix vulnerabilities
npm audit fix
# Force fix (may include breaking changes)
npm audit fix --force
# View detailed audit report
npm audit --json > audit-report.json
# Only check production dependencies
npm audit --production
Best Practices
  • Lock your versions

    Commit package-lock.json to ensure identical installs across environments.

  • Audit regularly

    Run npm audit before every release and in CI/CD pipelines.

  • Pin critical deps

    Use exact versions for security-critical packages: "bootstrap": "5.3.3"

  • Review before updating

    Check changelogs before major version updates. Test in a branch first.
