Button Security Best Practices

XSS prevention, CSP compliance, safe event handling, and content sanitization.

1. XSS Prevention in Button Content

VULNERABLE (DO NOT USE):
// DANGEROUS - XSS vulnerable
btn.innerHTML = userInput; // If userInput = '<img src=x onerror=alert(1)>'
// The malicious script executes!
SAFE: Using textContent
// SAFE - textContent escapes HTML
btn.textContent = userInput; // Renders as literal text, not HTML

2. Safe Dynamic Button Creation

VULNERABLE (DO NOT USE):
// DANGEROUS - HTML string injection
container.innerHTML = '<button class="btn">' + userName + '</button>';
// If userName = '<script>evil()</script>', code executes
SAFE: Using createElement + textContent

3. CSP-Compliant Event Handling

These buttons use data-action attributes and addEventListener instead of inline onclick. This works with strict CSP that blocks 'unsafe-inline'. 

<!-- CSP-safe pattern -->
<button data-action="greet" data-param="World">Greet</button>

<script>
  document.addEventListener('click', function(e) {
    const btn = e.target.closest('[data-action="greet"]');
    if (btn) greet(btn.dataset.param);
  });
</script>

6. CSRF Token Pattern

Forms with action buttons should include CSRF tokens to prevent cross-site request forgery. The token is validated server-side.

7. Double-Submit Prevention

Disable the submit button immediately on click to prevent accidental or malicious double-submission. Re-enable after server response.
