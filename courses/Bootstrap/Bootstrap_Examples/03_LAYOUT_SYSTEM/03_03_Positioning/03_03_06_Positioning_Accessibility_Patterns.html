Positioning & Accessibility

Positioning can disconnect the visual order from the DOM order, causing issues for screen readers and keyboard navigation. Here's how to do it right.

1. Visual Order vs DOM Order

Bad: Position Breaks Order
A (DOM 1st, visually 2nd)
B (DOM 2nd, visually 3rd)
C (DOM 3rd, visually 1st)

Screen readers read A-B-C, but eyes see C-A-B. Keyboard users tab through A-B-C order, which is confusing.

Good: DOM Matches Visual
1. First item (DOM and visual match)
2. Second item
3. Third item

DOM order matches visual order. Screen readers and keyboard navigation are predictable and logical.

2. Focus Order & Tab Navigation

Tab order follows DOM order, not visual order. Positioning can make tabbing unpredictable.

Test: Tab Through These Elements

Try tabbing: focus follows DOM order (1-2-3) not visual left-to-right reading order. If your visual layout suggests a different tab order, adjust the DOM order to match.

3. Fixed Elements & Content Hiding

Problem: Fixed elements (navbars, footers) can cover content for users who zoom in or use screen magnifiers. Always account for fixed element sizes. 
<!-- Fixed navbar that may hide content when zoomed -->
<nav class="navbar fixed-top">...</nav>

<!-- SOLUTION: Add padding to body equal to navbar height -->
<body style="padding-top: 56px;">
  <!-- Content won't be hidden behind navbar -->
</body>

<!-- Fixed bottom element -->
<div class="fixed-bottom bg-dark py-3">...</div>

<!-- SOLUTION: Add bottom padding -->
<body style="padding-bottom: 70px;">
  <!-- Content won't be hidden behind fixed footer -->
</body>

4. Overlays & Focus Management

Accessible Modal/Overlay Pattern
<!-- Modal with proper accessibility -->
<div class="position-fixed top-0 start-0 w-100 h-100"
     style="background: rgba(0,0,0,0.5); z-index: 1040;"
     role="dialog"
     aria-modal="true"
     aria-labelledby="modalTitle">

  <div class="position-absolute top-50 start-50 translate-middle">
    <div class="bg-white rounded p-4">
      <h2 id="modalTitle">Modal Title</h2>
      <p>Content here</p>
      <button>Close</button>
    </div>
  </div>
</div>

5. Screen Reader Considerations

Off-Screen Positioning

Elements positioned off-screen but still in DOM are read by screen readers.

/* Visually hidden but accessible */
.visually-hidden {
  position: absolute;
  width: 1px;
  height: 1px;
  padding: 0;
  margin: -1px;
  overflow: hidden;
  clip: rect(0, 0, 0, 0);
  border: 0;
}

/* OR use Bootstrap's class */
<span class="visually-hidden">
  Screen reader text
</span>
Absolute Positioned Labels

Labels positioned with absolute must still be associated with inputs.

<!-- Correct: label associated with input -->
<div class="position-relative">
  <label for="email"
         class="position-absolute"
         style="top: -20px;">
    Email
  </label>
  <input id="email" type="email">
</div>

<!-- WRONG: visual label, no association -->
<div class="position-absolute">Email</div>
<input type="email"> <!-- Missing id/for -->

6. Positioning Accessibility Checklist

CheckRequirement
☑DOM order matches visual reading order
☑Tab/focus order is logical and predictable
☑Fixed elements don't hide content when zoomed
☑Body padding accounts for fixed navbar/footer height
☑Overlays trap focus and return focus on close
☑Positioned labels are associated with form inputs
☑Absolutely positioned decorative images have empty alt
☑Sticky/fixed elements have appropriate z-index stacking
☑Use visually-hidden for screen-reader-only content
