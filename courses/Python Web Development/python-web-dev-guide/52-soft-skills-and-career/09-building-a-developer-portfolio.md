# Building a Developer Portfolio

## What You'll Learn

- What hiring managers actually look for (not what tutorials say)
- The three-project rule for portfolios
- How to write project descriptions that show problem-solving
- GitHub profile optimization
- Writing a technical blog as a portfolio piece

## Prerequisites

This is about career skills—no specific technical prerequisites.

## What Actually Gets You Hired

The biggest mistake developers make with portfolios is building what they think employers want vs. what actually matters:

```
┌─────────────────────────────────────────────────────────────────────────────┐
│                    WHAT HIRING MANAGERS LOOK FOR                            │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                             │
│  WHAT TUTORIALS TELL YOU:                                                 │
│  • Flashy side projects                                                   │
│  • Many technologies                                                      │
│  • Fancy animations                                                       │
│                                                                             │
│  WHAT ACTUALLY MATTERS:                                                   │
│  ✅ Can you ship?                                                        │
│  ✅ Can you solve problems?                                              │
│  ✅ Do you write maintainable code?                                      │
│  ✅ Can I work with you?                                                 │
│                                                                             │
│  Your portfolio should demonstrate:                                        │
│  • Code quality (not quantity)                                           │
│  • Problem-solving (show the problem AND solution)                        │
│  • Communication (clear writing)                                          │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘
```

## The Three-Project Rule

Three strong projects beat ten mediocre ones:

```
┌─────────────────────────────────────────────────────────────────────────────┐
│                    THE THREE-PROJECT RULE                                    │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                             │
│  Focus on 3 projects that show DEPTH:                                      │
│                                                                             │
│  Project 1: API + Database                                                 │
│  - Shows you can build backends                                            │
│  - Include database design, migrations                                     │
│                                                                             │
│  Project 2: Something with Complexity                                       │
│  - Caching, background jobs, real-time                                     │
│  - Shows you understand trade-offs                                         │
│                                                                             │
│  Project 3: Something You Care About                                      │
│  - Personal interest project                                               │
│  - Shows passion and motivation                                            │
│                                                                             │
│  Quality over quantity.                                                    │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘
```

## Writing Project Descriptions

The biggest mistake: describing what you built instead of what problem you solved.

```
┌─────────────────────────────────────────────────────────────────────────────┐
│                    PROJECT DESCRIPTION FRAMEWORK                            │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                             │
│  ❌ BAD DESCRIPTION:                                                       │
│  "Built a REST API with FastAPI, PostgreSQL, and Redis.                  │
│   Implemented JWT authentication."                                        │
│                                                                             │
│  This describes what you did, not why it matters.                         │
│                                                                             │
│  ✅ GOOD DESCRIPTION:                                                      │
│  "Built an AI assistant that answers questions about user's PDFs.         │
│  Users upload documents, the app extracts text, creates embeddings,      │
│  and uses GPT-4 to answer questions with citations.                       │
│                                                                             │
│  Tech: FastAPI, PostgreSQL (pgvector), Redis, OpenAI API                  │
│                                                                             │
│  Problem solved: Users had hundreds of PDFs they couldn't search.          │
│  Solution: Semantic search over document contents.                        │
│                                                                             │
│  This shows:                                                              │
│  • You can build complex systems                                           │
│  • You understand the problem domain                                       │
│  • You made trade-offs (chose pgvector for vector search)                 │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘
```

## GitHub Profile Optimization

Your GitHub profile is your business card. Optimize it:

### The Profile README

Create a `README.md` in a repo named exactly as your username:

```markdown
# Hi, I'm [Name] 👋

## I'm a Backend Engineer focused on Python and API Development

### 🔭 I work on
- Building APIs with FastAPI and Django
- Database design with PostgreSQL
- Cloud infrastructure on AWS

### 🌱 I’m currently learning
- Rust
- Kubernetes

### 📫 How to reach me
- Twitter: @yourhandle
- Email: you@example.com

### ⚡ Fun fact
I love making complex systems simple.
```

## Writing a Technical Blog

Blog posts are powerful portfolio pieces because they demonstrate:

- You understand a topic deeply enough to teach it
- You can communicate clearly
- You're passionate about learning

### Topics That Work

```
┌─────────────────────────────────────────────────────────────────────────────┐
│                    BLOG TOPICS THAT WORK                                    │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                             │
│  ✅ Tutorial based on something you built                                  │
│    "How I built a RAG system with FastAPI and pgvector"                  │
│                                                                             │
│  ✅ Problem-solution format                                                │
│    "Fixing slow PostgreSQL queries: A case study"                         │
│                                                                             │
│  ✅ Learning in public                                                     │
│    "What I learned building my first API"                                 │
│                                                                             │
│  ✅ Comparison/analysis                                                    │
│    "FastAPI vs Django: Which should you choose?"                          │
│                                                                             │
│  ❌ "How to do X" tutorials (already exist everywhere)                   │
│  ❌ Anything you don't have hands-on experience with                       │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘
```

### Where to Blog

| Platform | Best For |
|----------|----------|
| Dev.to | Discovery, community |
| Hashnode | SEO, custom domain |
| Personal blog | Full control |
| Medium | Reach (but mixed reputation) |
| GitHub Pages | Technical audience |

- Easy to customize themes
- Export to PDF, HTML

## Real-World Application

### At a Startup

At a startup, your portfolio matters less than:
- Can you ship?
- Can you figure things out?
- Are you easy to work with?

Your projects show you can ship. Your blog shows you can learn.

### At a Big Company

At big tech, your portfolio matters even less. They'll:
- Check your GitHub briefly
- Focus on interviews
- Ask about past experience

But a strong portfolio still helps:
- Get past initial screening
- Show specific skills they need
- Start conversations in interviews

## What NOT to Put in a Portfolio

```
┌─────────────────────────────────────────────────────────────────────────────┐
│                    AVOID IN PORTFOLIOS                                      │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                             │
│  ❌ Cloned tutorials                                                       │
│    "I followed a tutorial and built Uber"                                │
│                                                                             │
│  ❌ Tutorial code without customization                                   │
│    The exact same code as every other tutorial                            │
│                                                                             │
│  ❌ Outdated projects                                                      │
│    Django 2.0 projects in 2025                                            │
│                                                                             │
│  ❌ No documentation                                                       │
│    Repos with no README                                                    │
│                                                                             │
│  ❌ Excuses                                                                │
│    "This isn't finished because..."                                       │
│                                                                             │
│  ✅ Show what you actually built                                           │
│  ✅ Show what you learned                                                  │
│  ✅ Be honest about scope                                                  │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘
```

- Focus on 3 quality projects over 10 mediocre ones
- Describe problems solved, not just technologies used
- Optimize your GitHub profile with a README
- Write technical blog posts to demonstrate understanding
- Quality over quantity everywhere
