# Fallback Pattern
## Overview
The Fallback pattern provides an alternative response when an operation fails, allowing applications to maintain functionality even when primary services are unavailable. A fallback is a predefined alternative that returns degraded but still useful results instead of propagating errors to users. This pattern is essential for building resilient systems that can gracefully degrade under failure conditions. When a primary operation fails due to service unavailability, timeouts, or errors, the system automatically executes a fallback strategy that provides alternative functionality. This ensures continuous service availability and improves user experience even during partial system failures.
### Key Concepts
**Default Responses**: Returning predefined default values when services fail. This is useful for non-critical features where approximate data is acceptable. **Cached Responses**: Using previously cached data when fresh data cannot be retrieved. This is particularly effective for data that doesn't change frequently. **Alternative Service Execution**: Calling a secondary service endpoint when the primary service is unavailable. **Graceful Degradation**: Reducing functionality while maintaining core features when full functionality is not available.
### When to Use Fallbacks
1. Remote service is temporarily unavailable
2. Timeout occurs while waiting for response
3. Circuit breaker is open
4. Rate limit exceeded
5. Service returns an error
6. Network connectivity issues

import java.time.Instant;
import java.time.Duration;
import java.util.Map;import java.util.concurrent.ConcurrentHashMap;
public class ProductCache {    private final Map<String, CacheEntry<Product>> cache = new ConcurrentHashMap<>();    private static final Duration CACHE_TTL = Duration.ofMinutes(30);        public Optional<Product> get(String productId) {        CacheEntry<Product> entry = cache.get(productId);        if (entry == null) {            return Optional.empty();        }        if (entry.isExpired()) {            cache.remove(productId);            return Optional.empty();        }        return Optional.of(entry.getValue());    }        public void put(String productId, Product product) {        cache.put(productId, new CacheEntry<>(product));    }        public List<Product> getByCategory(String category) {        return cache.values().stream()            .filter(e -> !e.isExpired())            .map(CacheEntry::getValue)            .filter(p -> category.equals(p.getCategory()))            .toList();    }        public Optional<Product> getOrFetch(String productId, Supplier<Product> fetcher) {        return get(productId)            .or(() -> {                try {                    Product product = fetcher.get();                    put(productId, product);                    return Optional.of(product);                } catch (Exception e) {                    return Optional.empty();                }            });    }        private static class CacheEntry<T> {        private final T value;        private final Instant timestamp;                CacheEntry(T value) {            this.value = value;            this.timestamp = Instant.now();        }                T getValue() {            return value;        }                boolean isExpired() {            return Duration.between(timestamp, Instant.now()).compareTo(CACHE_TTL) > 0;        }    }
}  ```  ### Default Response Fallback Pattern ```java

Key benefits include:  - **Improved Availability**: Users experience uninterrupted service even when primary services fail - **Better User Experience**: Meaningful fallback responses maintain application usability  - **Reduced Error Propagation**: Prevents errors from cascading through the system - **Resource Efficiency**: Reduces unnecessary retry attempts and resource consumption  - **Operational Stability**: Systems remain stable under failure conditions  Effective fallback implementations combine cached data, default responses, and alternative service strategies to provide the best possible user experience during failures.  ## Best Practices 1. **Provide Meaningful Fallbacks**: Return useful data rather than generic error messages. 2. **Monitor Fallback Usage**: Track when fallbacks are triggered to identify problematic services. 3. **Implement Fallback Chains**: Have multiple fallback strategies in order of preference. 4. **Set Appropriate Cache TTLs**: Balance data freshness with availability requirements. 5. **Log Fallback Triggers**: Include sufficient context for debugging why fallbacks were used. 6. **Test Fallback Scenarios**: Regularly test fallback behavior in various failure conditions. 7. **Avoid Cascading Failures**: Ensure fallbacks themselves are resilient and don't call other failing services. 8. **Use Circuit Breakers**: Combine fallbacks with circuit breakers for comprehensive resilience. 9. **Consider User Notification**: Inform users when they're seeing fallback data. 10. **Keep Fallbacks Simple**: Fallback logic should be fast and not complex.
