Web Workers

Web Workers run JavaScript in background threads without blocking the UI.

Property/MethodDescription
new Worker(script)Create new worker
postMessage(data)Send message to worker
onmessageReceive message from worker
terminate()Stop worker

Creating a Web Worker

// main.js
var worker = new Worker('worker.js');

worker.onmessage = function(e) {
    console.log('Message:', e.data);
};

worker.postMessage('Hello');

// worker.js
self.onmessage = function(e) {
    var result = doHeavyCalculation(e.data);
    self.postMessage(result);
};

Service Workers

Service Workers act as a network proxy, enabling offline experiences and push notifications.

EventDescription
installWorker installed
activateWorker activated
fetchNetwork request
pushPush notification
syncBackground sync

Service Worker Lifecycle

// sw.js - Service Worker
self.addEventListener('install', event => {
    console.log('Service Worker installing');
    // Open cache and add files
    caches.open('my-cache').then(cache => {
        cache.addAll(['/', '/index.html', '/styles.css']);
    });
});

self.addEventListener('activate', event => {
    console.log('Service Worker activated');
});

self.addEventListener('fetch', event => {
    event.respondWith(
        caches.match(event.request)
            .then(response => response || fetch(event.request))
    );
});

Cache Strategies

StrategyDescription
Cache FirstCheck cache, then network
Network FirstTry network, fallback to cache
Stale While RevalidateUse cache, update in background
Cache OnlyOnly use cached responses
Network OnlyOnly use network

Best Practices

  • Use HTTPS for Service Workers
  • Keep worker scripts small
  • Use versioned caches
  • Clean up old caches on activate
  • Handle errors gracefully
