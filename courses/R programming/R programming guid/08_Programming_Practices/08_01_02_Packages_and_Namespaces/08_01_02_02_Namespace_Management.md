# Namespace Management in R

## Learning Objectives

- Understand R namespaces
- Use import and export directives
- Manage package dependencies
- Avoid namespace conflicts
- Use ::: for internal functions

## Theory

Namespaces in R prevent naming conflicts between packages. Each package has its own namespace. Exported functions are available to users; internal functions are hidden. The NAMESPACE file controls imports and exports.

Use roxygen2 tags (@export, @importFrom) to manage namespaces automatically. Avoid using ::: for external package functions.

## Step-by-Step

1. Add @export tag to functions users can access
2. Use @importFrom to import specific functions
3. Use @import to import entire packages
4. Run document() to regenerate NAMESPACE
5. Test for conflicts

## Code Examples

### NAMESPACE File

```r
cat("===== NAMESPACE FILE =====\n\n")

namespace_content <- "# Generated by roxygen2
exportPattern("^[[:alpha:]]+")
export(sum_stats)
export(plot_dist)
importFrom(ggplot2,.data)
importFrom(dplyr,mutate)
importFrom(dplyr,filter)
"

cat(namespace_content)
```

### Roxygen2 Export Tags

```r

#' Calculate summary statistics

  # ... implementation
}

#' Internal function (not exported)

  # ... implementation

```

### Importing Specific Functions

```r
cat("\n===== IMPORTS =====\n\n")

#' @importFrom dplyr mutate filter select
#' @importFrom ggplot2 ggplot aes geom_point

library(dplyr)
library(ggplot2)
```

### Avoiding Conflicts

```r
cat("\n===== AVOIDING CONFLICTS =====\n\n")

# Using package::function() syntax
result <- dplyr::filter(mtcars, cyl == 4)
result <- stats::filter(mtcars$mpg, rep(1, 3))

# Disambiguate with conflicted package
cat("# library(conflicted)\n")
cat("# conflicted::conflict_scout()\n")
cat("# conflict_prefer('filter', 'dplyr')\n")
```

### Managing Multiple Packages

```r
cat("\n===== MULTIPLE PACKAGES =====\n\n")

# At top of script
library(ggplot2)
library(dplyr)
library(tidyr)

# Disambiguate if needed
df <- ggplot2::select(mtcars, mpg, cyl)
df <- tidyr::pivot_longer(df, everything())
```

## Best Practices

1. Always use @export for public functions
2. Import specific functions, not entire packages
3. Use package::function() for disambiguation
4. Avoid ::: for external packages
5. Keep internal functions internal
6. Test for namespace conflicts
7. Document all imports
